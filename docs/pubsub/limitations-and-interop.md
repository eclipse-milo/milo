# Limitations and Interop

This page is the honest map of what Milo's PubSub modules implement, what they reject, and what they silently ignore. Read it before you depend on a Part 14 feature — and especially before you debug a config option that seems to do nothing.

Three modules are covered: `org.eclipse.milo:milo-sdk-pubsub` (the core config model, runtime, UADP and JSON codecs, message security, and built-in UDP transport), `org.eclipse.milo:milo-sdk-pubsub-mqtt` (the MQTT transport), and `org.eclipse.milo:milo-sdk-pubsub-server` (the `OpcUaServer` integration, including the opt-in SKS server face). The fourth artifact, `org.eclipse.milo:milo-sdk-pubsub-sks` (the SKS pull client), has no feature matrix of its own; its scope and limits are on the [message security page](message-security-and-sks.md).

## How to read the tables

Status means:

- **Works** — implemented and verified end-to-end.
- **Partial** — a usable subset works; the notes say exactly which part.
- **Config-only** — the configuration model expresses and round-trips it, but the runtime never executes it.
- **Absent** — no implementation exists.

"Behavior if configured" tells you what happens when you configure something the runtime does not do. *Rejected* means you get an error (the name is given — look for it in the exception or in diagnostics `lastError`). *Silently inert* means the config is accepted and ignored; nothing warns you. Inert options are the ones worth memorizing, because they fail without a trace.

## Message security and SKS

Message security and SKS key distribution are implemented; the full story — policies and modes, configuring secured groups, key providers, the key lifecycle, the receive-mode matrix, the security counters, and the opt-in SKS server face — has [its own page](message-security-and-sks.md). What belongs here is the boundary of what executes:

- **UADP only.** Both Part 14 policies (PubSub-Aes128-CTR, PubSub-Aes256-CTR) and both secured modes work, publishing and subscribing, on UDP and on UADP-over-MQTT — with the caveat that the security path is transport-independent (the secured bytes sit inside the NetworkMessage) and the in-repo secured test matrix runs over UDP, so secured-over-MQTT is applied-but-unverified, like MQTT TLS. JSON NetworkMessages have no message security in OPC UA 1.05 (Part 14 7.3.4.1): an enabled secured JSON-mapped group is rejected with `Bad_ConfigurationError`, and MQTT TLS is the JSON-side substitute.
- **The gate moved and changed name.** Where enabled secured groups used to fail with `Bad_NotSupported`, an enabled secured group now starts — provided it has a resolvable `SecurityGroupRef`, a supported (or null) security policy, and a `SecurityKeyProvider` bound via `PubSubBindings.securityKeys(...)`; anything missing is rejected with `Bad_ConfigurationError` naming the missing piece, at startup, reconfigure validation, and activation. Disabled groups are still tolerated for round-trip. Binding a provider to an unknown security group name still throws `IllegalArgumentException` at create time.
- **Pull only.** Key distribution is `GetSecurityKeys` pull — a Table 40 client (`SksSecurityKeyProvider`, in `milo-sdk-pubsub-sks`), a pre-shared-key provider (`StaticSecurityKeyProvider`), and an opt-in server face in `milo-sdk-pubsub-server`. Push (`SetSecurityKeys`) is not implemented, and neither are the SKS management methods (`AddSecurityGroup` and friends).
- **Receive side is mode-gated and counted.** A received message's security mode must be at least the reader group's configured mode (received Sign satisfies a Sign group, received SignAndEncrypt satisfies both secured modes); everything else — including any secured message to a mode-None group — is dropped and ticks `staleKeyMessages`, quietly. Signature verification happens before any payload byte is parsed and before the sequence-number window. Secured traffic matching no local reader at all is skipped without error or counter, as before.
- **The reader-level override is config-only.** `DataSetReaderConfig.messageSecurity` round-trips and its reference is validated, but the runtime resolves security at group level only.
- **Discovery and metadata stay mode None** in both directions, regardless of group security.

Five security counters (`encryptionErrors`, `decryptionErrors`, `invalidSignatureMessages`, `unknownTokenMessages`, `staleKeyMessages`) joined the diagnostics snapshot; their exact tick sites and quiet-versus-loud posture are on the [message security page](message-security-and-sks.md#the-key-lifecycle-at-runtime). Security drops never tick `decodeErrors`.

## Delta frames

`keyFrameCount` is honored, on both mappings. A writer emits a key frame on its first cycle after start or group activation and on every keyFrameCount-th publishing interval after that; the cycles between carry delta frames holding only the fields that changed since the last transmitted message (changed = the mask-projected wire value differs — a timestamp-only change counts only when timestamps are actually transmitted). A cycle in which nothing changed sends nothing at all: no empty delta, no sequence number consumed. Configure `keepAliveTime` if subscribers need a liveness signal across those quiet spells — quiet Operational writers now emit keep-alives (see the table above). Values 0 and 1 both mean every frame is a key frame (0 is the spec's non-cyclic value, which Milo does not model; the runtime clamps it). A delta that would carry every field is emitted as a key frame instead and resets the cadence. On the wire, UADP delta frames are DataSetFlags2 message type 0001 with per-field FieldIndex values (Part 14 Table 164); JSON delta frames are MessageType `ua-deltaframe` with a changed-fields-only payload (Table 185).

Two configurations cannot express delta frames and are rejected with `Bad_ConfigurationError` at startup and reconfigure (disabled components tolerated): a JSON writer with `keyFrameCount > 1` requires the DataSetMessageHeader in the group's effective network-message mask and the MessageType member in its DataSetMessage mask (Annex A.3.3.4); a UADP writer with `keyFrameCount > 1` cannot combine with a non-zero `ConfiguredSize` (fixed-size layouts are key-frame-only, Annex A.2.1.7). A writer that slips past validation — enabled after startup — degrades safely to every-cycle key frames. Custom mapping providers shadowing `uadp` or `json` own their wire format and are never second-guessed by these checks.

Receiving is unchanged: delta frames are decoded, for both UADP and JSON (`ua-deltaframe`), with per-field wire indices resolved against metadata. They are delivered to listeners, counted in diagnostics, reset the receive timeout, and recover a reader from Error — but they do not complete reader startup. A reader stays PreOperational until its first key frame or event message (a Part 14 SHALL): listeners see honest partial state from pre-baseline deltas while the reader's state says "no baseline yet", and the publisher's bounded key-frame cadence keeps the wait finite.

## Events

Event publishing has no API surface: there is no `PublishedEventsConfig`, and the encoders reject the event message kind with `Bad_NotSupported` (nothing in the publish path can produce one). This is a deliberate non-goal of this version, not an oversight. Importing a Part 14 wire config whose PublishedDataSet uses a `PublishedEventsDataType` source is rejected with `PubSubConfigValidationException` ("only PublishedDataItems dataset sources are supported").

Subscribing works: event DataSetMessages (UADP type `0x02`, JSON `ua-event`) are decoded and delivered like key frames, and an event message completes reader startup the same way a key frame does.

## RawData and promoted fields

UADP `RawData` field encoding and PromotedFields emission are absent, and they now fail fast: an enabled UADP-mapped writer group with the PromotedFields network-message mask bit, or an enabled writer with the RawData field-content mask bit, is rejected with `Bad_NotSupported` at startup and at reconfigure validation — before anything changes. The same checks run again at group and writer activation, so a component enabled after startup fails into PubSubState Error with `Bad_NotSupported` instead of running. The old trap — startup succeeds, the encoder rejects every publish cycle, and the writer looks healthy while sending nothing — is gone; the per-cycle encoder rejection remains only as a backstop for code driving the mapping API directly. Disabled components are tolerated so configs round-trip, and custom providers registered under the `uadp` mapping name are exempt (they own their wire format and may support these features).

On decode the behavior is tolerant: a RawData DataSetMessage is surfaced with `valid=false` and no fields (header fields intact), and a promoted-fields block is skipped over so the rest of the payload still decodes (the promoted values themselves are dropped).

JSON is unaffected: there the `RawData` mask bit is a Variant-representation modifier per Part 14, implemented and tested, not a rejection.

## Chunking and message size

NetworkMessage chunking is inbound-only: chunked NetworkMessages from peers are reassembled, but Milo never emits chunks — oversized messages are never split. `maxNetworkMessageSize` is enforced as a skip-or-deliver budget on both sides.

On a writer group, a non-zero `maxNetworkMessageSize` is enforced at publish time on every transport: a NetworkMessage whose encoded size exceeds it is skipped — never sent — with a `Bad_EncodingLimitsExceeded` diagnostics error at the group path carrying the actual and maximum sizes. Skipped DataSetMessages are not counted as sent, a perpetually skipped writer lapses into keep-alives (if configured), and a delta writer whose message was skipped re-baselines with a key frame on its next cycle. The default changed from 1400 to 0, meaning no enforced limit; UDP writer groups with a value above 65535 fail startup and reconfigure validation with `Bad_ConfigurationError` (the Part 14 UDP cap). Discovery messages (fixed 4096-byte budget) and broker metadata publishing are exempt from the group budget.

On a reader group, a non-zero `maxNetworkMessageSize` (default 0 = unrestricted) is honored on receive — a documented Milo extension; Part 14 gives the reader side no size semantics: a larger NetworkMessage is not seen by that group's readers and ticks `decodeErrors` at the group path with `Bad_EncodingLimitsExceeded`. The subscriber's UDP receive buffer is 65535 bytes (the Part 14 cap), so any datagram the OS can deliver arrives without truncation.

The fixed wire-format limits still stand, enforced at encode time with `Bad_EncodingLimitsExceeded`: more than 255 DataSetMessages in one NetworkMessage with a payload header, or any DataSetMessage body over 65535 bytes when the sizes array is present. Inbound chunked NetworkMessages from other stacks are reassembled: chunks may arrive out of order, each chunk is verified and decrypted individually when secured (per spec, every chunk is its own secured NetworkMessage), and the reassembled DataSetMessage re-enters decode and is delivered normally — a reassembled payload that then fails to decode ticks `decodeErrors` at the connection path. Reassembly carries hard, non-configurable DoS caps — a 4 MiB reassembled payload limit, at most 64 concurrent chunked streams (least-recently-active evicted), and 10 s idle eviction — and payloads dropped by a cap are logged, not counted. Two chunk cases still fail: chunked *discovery* messages are never reassembled — on the data path they tick `decodeErrors` with a `Bad_NotSupported` `lastError` at the connection, while the discovery socket ignores them like any other undecodable discovery traffic — and there is no chunk emission in the other direction. Other inbound decode failures (truncation included) tick `decodeErrors` the same way, while UADP still delivers whatever decoded before the failure point. Discovery metadata announcements that exceed the fixed, non-configurable 4096-byte discovery limit are dropped with a WARN and a `Bad_EncodingLimitsExceeded` diagnostics entry.

Practical consequence: keep datasets small enough that one NetworkMessage fits your transport MTU or broker limit — the stack will not fragment for you — and set the writer group's `maxNetworkMessageSize` to your transport budget so oversize messages fail loudly into diagnostics instead of at the OS or broker. If the group is secured, budget for the 46–47 bytes of [message security overhead](message-security-and-sks.md) per NetworkMessage; the limit is checked against the final secured size.

## Transports

Two transports exist: UDP (built into `milo-sdk-pubsub`, always available) and MQTT over TCP (`milo-sdk-pubsub-mqtt`, registered explicitly). Everything else is absent:

- **Ethernet (`opc.eth`) and AMQP** cannot even be expressed: `PubSubConnectionConfig` is sealed and permits only `UdpConnectionConfig` and `MqttConnectionConfig`. A Part 14 wire config with any transport profile URI other than udp-uadp, mqtt-uadp, or mqtt-json throws `PubSubConfigValidationException` on import.
- **MQTT over WebSocket**: `ws://` and `wss://` broker URIs are rejected with `Bad_NotSupported` when the connection opens; other unknown schemes get `Bad_ConfigurationError`.
- **UDP broadcast**: `UdpDatagramAddress` offers only `multicast` and `unicast` factories and the sockets never set `SO_BROADCAST`. One trap: a broadcast IP passed to `UdpDatagramAddress.unicast()` passes config validation and fails only when the OS rejects the send — this single case is silently inert rather than rejected.

Any enabled connection that no registered `TransportProvider` supports fails `startup()` with `Bad_ConfigurationError`. If you need another transport, the SPI is the supported route — see [Extending with the SPI](#extending-with-the-spi).

## Server integration limits

`ServerPubSub` (in `milo-sdk-pubsub-server`) is UDP-only. It builds its `PubSubServiceConfig` internally and exposes no way to register a transport provider, so MQTT connections in an attached config fail startup with `Bad_ConfigurationError` ("no TransportProvider supports connection") — or, if disabled at attach, fail with the same code the moment they are enabled. To publish or subscribe over MQTT, run a standalone `PubSubService` alongside the server instead. An options-level transport-provider hook is a known backlog item.

Remote configuration over OPC UA is the Part 14 *file model*, opt-in via `ServerPubSubOptions.allowRemoteConfiguration(true)`: the ns0 `PubSubConfiguration` file object's eight methods plus per-component `Enable`/`Disable` (see [server integration](server-integration.md#remote-configuration)). What remains absent: the deprecated per-element imperative methods (AddConnection, RemoveConnection, ...), the SKS management methods, and the dataset-level binding methods (CreateTargetVariables, CreateDataSetMirror, ...) are unbacked, and a client invoking them receives `Bad_NotImplemented` — as does everything else when remote configuration is off; the only ns0 PubSub method backed independently of it is `GetSecurityKeys` under the opt-in [SKS server face](message-security-and-sks.md#the-sks-server-face).

One interop caveat specific to the file model: no open-source OPC UA stack implements it as a client, so there is no automated third-party peer for it — Milo's implementation is contract-tested against the spec text, and manual smoke runs use commercial tooling (OPC Labs QuickOPC, UaExpert's generic FileType access). If your configuration tool speaks the file model, expect to be an early interop pairing.

## Standalone SubscribedDataSets and mirror

Standalone SubscribedDataSet support is partial: a `StandaloneSubscribedDataSetRef` on a reader resolves that reader's configured decode metadata, refs are validated at config build (unresolved refs are rejected), and chained refs resolve. The resolution is wired into the live decode path but, unlike reader-level metadata, is not covered by an end-to-end data-flow test in this release — treat it as applied-but-unverified, like MQTT TLS. What does not work: in `milo-sdk-pubsub-server`, TargetVariables carried *inside* a standalone SubscribedDataSet get no writer — attach succeeds, the targets are still validated, a WARN is logged, and then nothing is ever written. Only a reader-level `TargetVariablesConfig` drives writes.

`SubscribedDataSetMirror` is absent entirely. The typed API cannot express it (the sealed spec type permits only TargetVariables and standalone refs), and a `SubscribedDataSetMirrorDataType` arriving via Part 14 import is rejected with `PubSubConfigValidationException`.

## Timing offsets

The deterministic-timing offsets — `SamplingOffset`, `PublishingOffset`, `ReceiveOffset`, `ProcessingOffset` — have no typed fields in the [configuration model](configuration.md) and no runtime behavior. Publishing is plain fixed-rate scheduling with zero initial delay, and dataset snapshots are pulled inline at publish time; readers process messages on arrival.

Offsets arriving in an imported wire config are not rejected: the whole UADP message-settings structure is diverted into the raw `ExtensionObject` escape hatch (`rawMessageSettings`), preserved with round-trip fidelity, and even surfaced as info-model properties by the server module — but the runtime never decodes or consults those values. The raw hatch exists so configs survive a round trip through Milo, not so offsets take effect. If you author config in Java and need the offsets present for a peer's benefit, you must hand-encode the `UadpWriterGroupMessageDataType` / `UadpDataSetReaderMessageDataType` yourself via `rawMessageSettings`.

## Sequence numbers

The publisher emits sequence numbers per spec: a per-WriterGroup NetworkMessage sequence and a per-DataSetWriter DataSetMessage sequence, with keep-alives carrying the next expected number without incrementing it. The NetworkMessage sequence and the UADP DataSetMessage sequence wrap at UInt16; the JSON DataSetMessage sequence is genuinely UInt32 and rolls over at 2^32.

The subscriber now tracks them — always on, no config knob, both transports. Each reader keeps Part 14 7.2.3 recency windows per stream: per publisher and writer group for the NetworkMessage sequence (UADP only — JSON has none), per publisher and writer for the DataSetMessage sequence, with wildcard readers tracking each stream independently. Duplicates and reordered-older messages are dropped as stale; messages outside the window are dropped as invalid. Dropped means dropped: not delivered, no receive-timeout reset, no startup completion, no Error recovery — and never reordered into place, so a reordered message is lost, not delayed. This is a delivery-semantics change from earlier versions, which delivered duplicates; in particular, MQTT QoS 1 redelivery of sequence-numbered messages is now deduplicated. The per-reader counters `staleSequenceMessages` and `invalidSequenceMessages` tick in the diagnostics snapshot only (drops are normal operation: no `lastError`, no error event), and `DataSetReceivedEvent` now carries the wire's `networkMessageSequenceNumber` (UADP only) and `dataSetMessageSequenceNumber`. The drop counters are Milo vendor counters on the SDK API — Part 14 names no counter for sequence-window drops, so they have no ns0 counterpart in the server module's §9.1.11 exposure either.

The edges: messages without sequence numbers bypass the window entirely — every one counts as new, per spec. Keep-alives seed unseeded streams and reseed the window when their carried value is off-window (the publisher is authoritative about its own counter; this is how a publisher whose sequence restarted at 0 — a process restart, or a component removed and re-added by reconfiguration; path-stable reconfigure restarts preserve the sequence — recovers). Stream records are discarded after twice the `messageReceiveTimeout` of accepted-message or keep-alive silence; with the default timeout of 0 there is no time-based discard, and instead 16 consecutive rejections on a stream discard its record so the next message reseeds (WARN logged). Tracked streams are bounded per reader at 4096 per window kind — the NetworkMessage and DataSetMessage windows each keep their own map — with the least recently used stream evicted beyond that (WARN logged), a guard against rogue traffic fabricating stream identities. JSON MessageId-based deduplication is not implemented.

## The decoder accepts more than the encoder produces

A recurring pattern in the tables above: the decode path is a strict superset of the encode path. Receiving any of these from a third-party publisher is tolerated; producing them from Milo is impossible:

| Received thing | What happens on receive | Can Milo send it? |
|---|---|---|
| Event DataSetMessages | Decoded and delivered (complete reader startup) | No — no publish API |
| Chunked NetworkMessages | Reassembled (secured chunks verified/decrypted individually) and delivered; DoS-capped | No — no chunk emission |
| Chunked discovery messages | Never reassembled; on the data path: `decodeErrors` + `Bad_NotSupported` `lastError` | No — discovery is never chunked |
| RawData DataSetMessages (UADP) | Surfaced header-only with `valid=false` | No — config rejected up front (startup/reconfigure/activation) |
| PromotedFields blocks (UADP) | Skipped; rest of payload decodes | No — config rejected up front (startup/reconfigure/activation) |
| Sign-only messages in either Annex A nonce form | Verified and delivered (NonceLength is self-describing) | Milo emits the full-nonce form only |
| Secured messages carrying a SecurityFooter | Footer skipped; message verified and delivered | No — neither PubSub policy uses a footer |

Delta frames used to lead this table, and security-flagged messages sat in it for one release; Milo now sends both — see [Delta frames](#delta-frames) and [Message security and SKS](message-security-and-sks.md). What remains of the old security row: a secured message that matches no local secured reader group, or that fails its checks, is dropped and [counted](message-security-and-sks.md#the-key-lifecycle-at-runtime), never delivered.

The practical reading: Milo as a *subscriber* is tolerant of fuller-featured peers, but don't design a system that needs Milo to *publish* any of the above.

## Interop status

The stack has been manually interop-tested against a third-party Part 14 implementation, in both directions (Milo publishing / Milo subscribing), over UDP/UADP multicast and over MQTT with the JSON mapping. UDP unicast and UADP-over-MQTT have not been part of those third-party runs; they are verified by automated in-repo tests instead.

The harness used for those runs is `PubSubInteropTool`, a command-line tool in the test tree of `milo-sdk-pubsub-mqtt` (`org.eclipse.milo.opcua.sdk.pubsub.mqtt.PubSubInteropTool`). It drives the real `PubSubService` engine — or, in its "raw" modes, a bare socket or MQTT client, useful against a misconfigured or unknown peer — and prints everything sent and received. It offers publish, subscribe, and raw-capture modes for both UDP and MQTT, all emitting or consuming a fixed "MiloInterop" dataset, and its UDP modes take secured-UDP options (`--security-mode`, `--security-policy`, `--static-keys`/`--static-keys-hex`, or `--sks` to pull rotating keys from a live SKS) for exercising message security against a peer. Its MQTT modes also expose JSON status/Will testing: `mqtt-subscribe` prints remote publisher status events, `mqtt-status` watches only `<prefix>/json/status/<publisherId>` or `<prefix>/json/status/+`, and `mqtt-publish --status auto|disabled|will|cyclic` selects the publisher status policy under test. Its class Javadoc contains the exact build and run commands plus worked examples; running it with no arguments prints full usage. If you need to validate Milo against your own Part 14 stack, that tool is the intended starting point.

Message security's verification status, separately: the secured encode path is proven in-repo against hand-derived golden vectors with an independent (non-JVM) recompute harness, and the full mode-by-policy matrix runs as Milo-to-Milo loopback tests, including an end-to-end SKS scenario (embedded SKS server, secured publisher and subscriber, live key rotation). Secured runs against third-party stacks go through the tool's secured flags; captured third-party secured vectors are not yet part of the automated evidence.

Remote configuration's interop status, likewise: the Part 14 file model has no open-source client-side peer, so its automated evidence is Milo-driving-Milo (client-driven, over real service calls) against the spec-anchored contract; third-party verification is a manual exercise with commercial tooling (OPC Labs QuickOPC, UaExpert's generic FileType dialog). The status-event bridge's abstract-typed notifications (`OfType(i=15535)`) are likewise verified Milo-to-Milo; check your client's tolerance for abstract event types before relying on them.

## Extending with the SPI

If you need a transport this version doesn't ship — AMQP, Ethernet, anything — the `TransportProvider` SPI is the proven path: the entire MQTT module is implemented through it, with no special access to engine internals. Custom message mappings plug in the same way via `MessageMappingProvider`. Registration is explicit and programmatic; there is no `ServiceLoader` discovery:

```java
// MQTT is NOT auto-discovered: registering MqttTransportProvider on the service config is
// the one line that wires the MQTT transport in. Without it, startup fails with
// Bad_ConfigurationError ("no TransportProvider").
PubSubServiceConfig serviceConfig =
    PubSubServiceConfig.builder().transportProvider(MqttTransportProvider.create()).build();
```

(from `MqttUadpPublisherExample` in `milo-examples/pubsub-examples`)

Three things to know before implementing a provider:

- Selection is by `supports(connection)` only — the first registered provider that returns true wins, then the built-in UDP transport is the fallback. `transportProfileUri()` is declarative and never consulted for selection. A config referencing a connection no provider supports fails startup (and reconfigure validation) with `Bad_ConfigurationError`.
- A custom message mapping needs more than a provider: the engine derives the mapping name from the message-settings *class* (built-in `Uadp*Settings` map to `uadp`, `Json*Settings` to `json`, anything else to the settings class's simple name), so a custom mapping also requires custom `WriterGroupMessageSettings` / `DataSetWriterMessageSettings` / `DataSetReaderMessageSettings` types whose simple class name equals your provider's `mappingName()`. Registering a provider under the name `uadp` or `json` shadows the built-in on the data plane — supported and tested.
- The UADP discovery plane is exempt from both SPIs: discovery probes and announcements always go through the built-in UDP transport and built-in UADP mapping. Custom providers affect the data plane only.
- `MessageMappingProvider.encodeMetaData` is optional, and its default implementation throws `Bad_NotSupported`. Override it if your mapping will be used on broker connections — otherwise metadata publishing for those writers fails into diagnostics (`lastError` on the writer path) while data messages keep flowing, which is easy to miss.
